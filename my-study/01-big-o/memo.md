# アルゴリズムの複雑さ / Big-O記法

## なぜBig-Oが必要か

アルゴリズムの「速さ」を測りたい時、実際の実行時間（秒）で測るのは不適切。マシンの性能・言語・コンパイラに左右されるから。
そこで、入力サイズ `n` が増えた時に処理量がどう増えるかという「増え方の傾向」だけに注目する。

Big-O は入力が大きくになったときの最悪のケースの増加率の上限を表す記法。

### 「料理」と「探しもの」でイメージ

Big-Oが知りたいのは、たった一つだけ。

> 「データの量が増えたら、作業はどれくらい大変になるの？」

「友達が1人来たとき」と「友達が100人来たとき」で、あなたの手間がどう変わるか。考えることはこれだけ。

## 主要な計算量（速い順）

| 記法         | 名前         | n=1,000,000 のイメージ | 代表例                               |
|--------------|--------------|------------------------|--------------------------------------|
| `O(1)`       | 定数時間     | 1                      | 配列の添字アクセス `arr[i]`          |
| `O(log n)`   | 退数時間     | 約20                   | ソート済み配列の二分検索             |
| `O(n)`       | 線形時間     | 1,000,000              | 配列を1回なめる線形探索              |
| `O(n log n)` | 線形対数時間 | 約2000万               | マージソート, クイックソート（平均） |
| `O(n²)`      | 二乗時間     | 1兆                    | 二重ループ（バブルソート等）         |
| `O(2ⁿ)`      | 指数時間     | 天文学的               | 部分集合の全列挙、素朴なフィボナッチ |
| `O(n!)`      | 階乗時間     | 論外                   | 全順列の列挙                         |

### `O(1)`...量が増えても手間は変わらない（一番楽）

例え: 冷蔵庫から「一番上の段のジュース」を取る

中身が10本でも1000本でも、一番上を取るだけなので手間は同じ。
何個あっても一瞬で終わる。

```ts
// 何個入っていても、最初の1個を取るだけ
arr[0]
```

### `O(n)`...量が2倍になれば、手間も2倍（素直）

例え: 教室の全員に1枚づつプリントを配る

生徒が10人なら10回、100人なら100回配る。
人数と全く同じ回数だけ働く。

```ts
// 中身の数だけ繰り返す
for (const x of arr) {...}
```

> 👉 ポイント: 「全員を1回ずつ見る」タイプは `O(n)`。

### `O(n²)`...量が2倍になると手間は4倍（けっこうキツイ）

例え: クラス全員と全員を「総当たり」で握手させる

1人ずつが、他の全員と握手する。
人数が増えると、握手の回数は爆発的に増える（10人→約100回、100人→約10000回）。

これは「全員をなめるループの中で、さらに全員をなめるループ」が起きているとき。

```ts
for (...) {   // 全員分
  for (...) { // その中でまた全員分 
  }
}
```

> 👉 ポイント: ループの中にループがあると `O(n²)`。「重い処理」の代表。

### `O(log n)`...量が増えても手間はちょっとしか増えない（賢い）

例え: 辞書で「さ行」の単語を探す時、1ページずつ捲らず、真ん中を開き、探している単語より前か後ろかを判断し、いらない半分を捨て、また残りの真ん中を開く。

毎回、候補を半分に減らすので、ページ数が2倍になっても作業は「あと1回」増えるだけ。

> 👉 ポイント: 毎回半分に絞れるなら `O(log n)`。とても速い。

### 速さの感覚（楽な順）

1. `O(1)`
2. `O(log n)`
3. `O(n)`
4. `O(n²)`

## 大事なルール

1. 「だいたい何倍？」だけ見て、細かい数は捨てる。
2. 「一番重い作業」だけ見る。
